# ADR-015: リレーション系追加フォーム縦1列レイアウト

- **ステータス**: 承認
- **カテゴリ**: UI/UX
- **日付**: 2026-04-16
- **対象プロジェクト**: 05_MermaidAssist
- **関連ADR**: ADR-012 (DiagramModule v2)

## コンテキスト

v1.1.0 までの property panel において、「リレーション系」（メッセージ・エッジ・遷移・関連・リレーションシップ）の追加フォームは、From/Arrow/To 等のドロップダウンを **横並び** で配置していた。これにより以下の問題が発生:

- ラベル無しで From/To の識別困難（特に participant 名が "Client/Server" のような対称的なとき）
- 220px の property panel に対し3-4個の select が並ぶと最後の要素が **画面外に切れる**
- 個別要素編集パネルは最初から縦1列・ラベル付きで設計されていたため、追加と編集で **異なる UI を学習する必要** が生じていた

実ユースケースで Sequence 図のメッセージ送信先指定が直感的でないという指摘があり、Tier1 全5図形（Sequence/Flowchart/State/Class/ER）に共通する UX 問題として顕在化した。

## 検討した選択肢

### A) 縦1列・ラベル付きに統一（採用）
- 概要: 追加フォームを編集パネルと同じ縦1列レイアウトに揃え、From/Arrow/To/Card 等にラベル付与
- メリット: 追加と編集で同一の mental model、識別容易、画面外切れなし、`window.MA.properties.selectFieldHtml` ヘルパー流用で実装簡単
- デメリット: 縦方向のスクロール量が若干増える（許容範囲）

### B) 横並びのまま、各 select 上にラベル追加
- 概要: 既存横並びを維持しつつ「From」「To」のラベルだけ上に配置
- メリット: 縦方向コンパクト
- デメリット: 220px 幅では3-4個のラベル付き select は依然として狭く、画面外切れは解消できない

### C) SVG 上のクリック2回でリレーション作成（Connection Mode）
- 概要: ADR-013 の Connection Mode を全図形で実装し、ソース/ターゲットを直接クリックで指定
- メリット: 描いている感覚があり最も直感的
- デメリット: 各図形の SVG 構造解析と handler 付与が必要で実装複雑、初学者にはモード概念の学習コスト、フォーム経由の操作も並存させる必要がある（フォール バック）

## 決定

**A を全5図形に適用** (v1.2.0)。理由:

1. 既存の `selectFieldHtml` ヘルパーで実装が小さい（モジュールあたり 5-10行の置換）
2. 編集パネルと完全に同じレイアウト → 学習コスト一元化
3. element ID 不変で既存 E2E と互換維持
4. 5図形に同一パターンを適用することで保守性も向上

C は将来的な強化候補として残置（ADR-013 の枠組みは維持）。

## 結果

- 5モジュール（sequence/flowchart/state/class/er）の追加フォームを統一
- 5シナリオ（OAuth/受注フロー/組み込み状態/ECドメイン/DB設計）を MCP 実機で検証、全 PASS（console error 0）
- E2E テスト E15-E24 (10件) 全 PASS、既存 123 → 133 へ
- system-tester による評価設計でカバレッジ 100% を確認

想定外: 並列ワークツリーで同時改修した際、state.js のリファクタで `var P = ...` と `props.X` の alias 不整合（agent が現状の alias 名を確認せず props を使った）が発生し、E2E 4件失敗 → 即修正で復旧。

## 教訓

1. **横並び複数 select は避ける**: 220px のような狭い panel では3個以上の横並び select は必ず画面外切れを起こす。今後の relation 系 UI は最初から縦1列で設計すること
2. **追加と編集の UI レイアウトを揃える**: 同じ操作概念の UI が2形式あると学習コスト二重。再利用ヘルパーで形式を強制すべし
3. **並列リファクタでは既存 alias 名を変更しない**: 各モジュールが `var P = ...` か `var props = ...` のどちらを使っているかを **個別タスクの prompt で明示** すれば agent の選択ミスを防げた。横断 alias 命名統一を将来課題とする
4. **MCP 実機検証の威力**: 自動 E2E (133件 GREEN) でも、実ユースケース（例: 9メッセージのOAuthフロー）を流すと UX 上の自然さが確認できる。リリース判断には両方が必要
