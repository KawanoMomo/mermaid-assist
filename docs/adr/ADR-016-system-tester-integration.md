# ADR-016: system-tester エージェント連携によるテスト設計トレーサビリティ

- **ステータス**: 承認
- **カテゴリ**: エージェント運用
- **日付**: 2026-04-16
- **対象プロジェクト**: 05_MermaidAssist

## コンテキスト

開発ループは以下のサイクルで回している:

```
brainstorming → spec.md → writing-plans → plan.md → 実装（subagent） → 検証
```

このうち「検証」フェーズで、E2E テストや MCP 検証が行われるが、**仕様書の規範的記述（要件）と実際のテストコードの間にトレーサビリティが無い** という問題があった:

- 仕様書 §3.2 の「全5図形に適用する」要件に対応するテストはどれか？
- E15 のテストは仕様書のどの要件を検証しているのか？
- 仕様書に書かれた要件のうち、テストでカバーされていないものは無いか？

これらが手動チェックに依存しており、仕様変更時にテストが追従しているか追跡困難。また、判定基準に曖昧語（「正しく動作する」「適切に表示される」等）が混入するリスクもあった。

## 検討した選択肢

### A) system-tester エージェントを工程に組み込む（採用）
- 概要: ワークスペース共通の `.claude/agents/system-tester.md` を活用。仕様書 (.md) を入力に、要件→評価項目→テストタスクを構造的に抽出し、双方向トレーサビリティ表を含む test-spec.md を生成
- メリット: 仕様書からの自動抽出でカバレッジ100%を機械的に保証、禁止語（曖昧語）検出、未確定事項を明示、E2E テスト ID と評価ID を相互参照可能
- デメリット: 1工程追加、agent 出力の品質が opus モデルに依存

### B) 手動でトレーサビリティ表を maintain
- 概要: spec/plan/test 各ドキュメントに手動でリンクを書き合う
- メリット: 追加コスト低（最初は）
- デメリット: 仕様変更時に同期が崩れる、人為的な漏れ、曖昧語チェック不可

### C) 既存の brainstorming/writing-plans で代用
- 概要: 設計時に「どうテストするか」を spec/plan に書き込む
- メリット: 既存ワークフロー内で完結
- デメリット: 検証可能性の構造化（観測対象・判定基準の一意化）が弱く、トレーサビリティ表が無い

## 決定

**A を採用** し、以下のワークフローに組み込む:

```
brainstorming → spec.md
              ↓
   ┌──────────┴──────────┐
   ▼                     ▼
writing-plans       system-tester
   ↓                     ↓
plan.md              test-spec.md
   ↓                     │
実装 + E2E + MCP ←──────┘ (E2E ID と TC-ID を相互参照)
```

system-tester は **設計のみ行いテスト実行はしない** という責務分離を厳守。実行は subagent-driven-development や MCP 経由で別途実施。

実験的に v1.2.0 リリースで初導入。

## 結果

v1.2.0 「追加フォーム統一」spec に対し system-tester を実行:
- 26 要件抽出、32 評価項目、32 テストタスク生成
- カバレッジ自己検査: REQ→EV 26/26 (100%), EV→TC 32/32 (100%)
- 禁止語検出 0件、重複評価項目 0件
- 3件の未確定事項 (U-1: viewport baseline, U-2: console error の範囲, U-3: 中間スクリーンショット要否) を曖昧推測せず明示

良かった点:
- 「曖昧」が構造的に禁止される効果が大きい（"正しく" 等を機械的に弾ける）
- E15-E24 の E2E テストと EV-E15/E17/... が ID で対応付き、追跡容易
- 仕様書側に書かれていない要件を agent が推測補完しないため、spec 側の不足を可視化できる

想定外:
- ワークスペース共通の `.claude/agents/system-tester.md` をプロジェクトの `.claude/agents/` にコピーしないと subagent_type で認識されなかった（セッション開始時にしか agent registry がロードされないため）→ プロジェクトに同ファイルをコピーして commit することで解決

残課題:
- 未確定事項の解消フィードバックループ（U-1〜U-3 を spec に反映する手順）が未整備
- system-tester 出力の test-spec.md と実 E2E のずれを定期チェックする仕組みが無い

## 教訓

1. **エージェントは workspace 共通定義を project に明示コピー**: workspace 配下の `.claude/agents/*.md` は新規セッションでないと反映されない場合がある。プロジェクト依存を明示するためにも、使用するエージェントは `<project>/.claude/agents/` に commit すること
2. **「曖昧禁止」の構造化は強力**: 仕様書の judgment criteria に曖昧語を含めない方針は、機械チェック可能なので必ず採用すべし
3. **設計と実行の責務分離**: system-tester に「テスト実行」もさせると複雑化。設計エージェントは設計のみに留め、実行は別エージェント or 人間に委ねる
4. **未確定事項は推測で埋めない**: agent が `coverage_ok: true` を返しても unresolved_count > 0 ならば、spec に欠落があるサインとして spec 側に逆流させること
